# i18n

Use the current `next-intl` setup as the only i18n path in this repo.

## Current structure

```text
frontend/
├── messages/
│   ├── en-US/
│   │   ├── common.json
│   │   ├── navigation.json
│   │   └── ...
│   └── zh-CN/
│       ├── common.json
│       ├── navigation.json
│       └── ...
├── i18n/
│   └── request.ts
├── lib/i18n/
│   ├── index.ts
│   └── server.ts
└── providers/
    └── i18n-provider.tsx
```

## Rules

- Use `next-intl/plugin` in `frontend/next.config.mjs`.
- Keep request configuration in `frontend/i18n/request.ts`.
- Resolve locale from the existing locale cookie.
- Keep local messages in `frontend/messages/<locale>/*.json`.
- Split messages by feature/shared domain, not by page file and not in one giant locale file.
- Current shared domains should stay separate:
  - `common`
  - `navigation`
  - `metadata`
  - `errors`
- Use nested JSON messages and dotted lookup keys.
- Use the project `useI18n()` wrapper in client components where it already exists.
- Keep server-side locale helpers in `frontend/lib/i18n/server.ts`.
- Keep metadata localized through the same message files.
- Keep locale loading centralized in `frontend/lib/i18n/index.ts`, where per-locale modules are merged into one `next-intl` message object.

## Message key conventions

- Use feature namespaces:
    - `common.*`
    - `navigation.*`
    - `auth.*`
    - `users.*`
    - `dicts.*`
    - `menus.*`
    - `tenants.*`
    - `metadata.*`
- Prefer complete phrase keys over assembling mixed translated and hardcoded sentence fragments.
- When adding a new key, update every supported locale in the same change.
- When adding a new feature, add a new module file under every locale directory instead of extending unrelated message files.

## Do not

- Do not introduce a second custom i18n provider.
- Do not store local message catalogs as large TypeScript constant maps when JSON files already exist.
- Do not collapse all locale messages back into a single `en-US.json` / `zh-CN.json` file.
- Do not hardcode user-facing text in already-localized features.
- Do not create locale-prefixed routing unless the product requirement explicitly asks for it.
